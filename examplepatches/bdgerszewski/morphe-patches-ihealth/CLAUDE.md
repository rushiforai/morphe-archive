## Steps to produce a patch

### Step 1: Explore the app code with JADX

Use the `jadx` skill to decompile the XAPK and read the Java source. The APK is at `ihealth-context/iHealth+MyVitals_4.14.5.xapk`.

Useful strategies:
- Search for unique strings visible in the UI to locate the class and method that renders the target UI.
- Search for ad-related strings or class names (e.g. `"banner"`, `"ad"`, `"AdView"`) to find the ad banner logic.
- Once you find a candidate class, read the full class to understand the call graph — which method shows the view, what conditions control it, and whether there's a simple boolean/flag you can short-circuit.
- Look at the Smali view (not just the Java decompile) for the actual bytecode you'll need to match with fingerprints. JADX can show Smali alongside Java.

### Step 2: Fingerprint the target method

Fingerprints identify a method by its characteristics so the patch survives obfuscation across app updates. Create fingerprints in `patches/src/main/kotlin/app/template/patches/<feature>/Fingerprints.kt`.

**Pick fingerprint attributes that are stable across updates:**
- Unique string literals inside the method (`string("mmHg")`, `string("Text generated by AI")`)
- Non-obfuscated method calls on SDK/framework classes (`methodCall(definingClass = "Landroid/view/View;", name = "setVisibility")`)
- Return type and parameter types using JVM descriptors (`returnType = "V"`, `parameters = listOf("Ljava/lang/String;")`)
- Access flags (`AccessFlags.PUBLIC`, `AccessFlags.PRIVATE`, etc.)
- Specific opcodes at key points (`opcode(Opcode.RETURN_VOID)`)

**Avoid:**
- Obfuscated class or method names (they change each release)
- Opcode-only patterns (`OpcodesFilter`) unless nothing else works — they are fragile

**Fingerprint filters must appear in the same order as the instructions in the method.** Between filters, any number of unrelated instructions can exist. Use `MatchAfterImmediately()` when a filter must immediately follow the previous one.

Declare fingerprints as `object` (named class) rather than a local variable so the class name appears in exception stack traces if the match fails:

```kotlin
object AiReadingModalFingerprint : Fingerprint(
    returnType = "V",
    strings = listOf("Text generated by AI", "mmHg"),
    filters = listOf(
        methodCall(definingClass = "Landroid/view/View;", name = "setVisibility"),
    )
)
```

To narrow a fingerprint to a specific class, either use `definingClass = "Lcom/ihealth/..."` (or whatever the current package name is / starts with) or use a `classFingerprint` that finds the class via a unique method within it (e.g. a `toString` that references a unique field name).

### Step 3: Write the patch

Create the patch in `patches/src/main/kotlin/app/template/patches/<feature>/SomePatch.kt`.

**Patch skeleton:**

```kotlin
package app.template.patches.$APP_NAME$

import app.morphe.patcher.extensions.InstructionExtensions.addInstructions
import app.morphe.patcher.patch.bytecodePatch
import app.template.patches.shared.Constants.COMPATIBILITY_IHEALTH

val disableAiBroadcastPatch = bytecodePatch(
    name = "Disable AI broadcast modal",
    description = "Prevents the AI-generated reading modal from appearing after a blood pressure measurement.",
    default = true
) {
    compatibleWith(COMPATIBILITY_IHEALTH)

    execute {
        // Access the matched method via AiReadingModalFingerprint.method
        // Modify from the last instruction match index to the first to keep indexes stable
        AiReadingModalFingerprint.method.addInstructions(0, "return-void")
    }
}
```

**Common modification patterns:**

- **Short-circuit a void method** (prevent it from running at all):
  ```kotlin
  fingerprint.method.addInstructions(0, "return-void")
  ```

- **Short-circuit a boolean method to always return false** (e.g. "should show ads"):
  ```kotlin
  fingerprint.method.addInstructions(0, """
      const/4 v0, 0x0
      return v0
  """)
  ```

- **Override a boolean result after a method call** (e.g. the result of `shouldShow()` is in register v0):
  ```kotlin
  // instructionMatches indexes correspond to the filter list (0-based)
  val resultMatch = fingerprint.instructionMatches[1]
  val index = resultMatch.index
  val register = resultMatch.getInstruction<OneRegisterInstruction>().registerA
  fingerprint.method.addInstructions(index + 1, "const/4 v$register, 0x0")
  ```

- **When making multiple edits to the same method**, always edit from the last index to the first to avoid index drift. Or call `fingerprint.clearMatch()` then `fingerprint.match(...)` after each edit to get fresh indexes.

Add the `Compatibility` constant to `patches/src/main/kotlin/app/template/patches/shared/Constants.kt`:

```kotlin
val COMPATIBILITY_IHEALTH = Compatibility(
    name = "iHealth MyVitals",
    packageName = "com.ihealth.vitals",  // verify the real package name via JADX
    apkFileType = ApkFileType.XAPK,
    appIconColor = 0x00AEEF,
    targets = listOf(
        AppTarget(version = "4.14.5")
    )
)
```

### Step 4: Build the patch

```bash
./gradlew buildAndroid
```

This compiles the Kotlin patch source into a `.mpp` bundle at `patches/build/libs/patches-*.mpp`.

If the build fails:
- Check that fingerprint class/object names are exported and the patch `val` has `@Suppress("unused")`.
- Check that imports resolve — use the example patch `patches/src/main/kotlin/app/template/patches/example/` as reference.

### Step 5: Apply the patch with morphe-cli

```bash
java -jar /home/ben/repos/morphe-cli/build/libs/morphe-cli-1.9.1-all.jar \
  patch \
  --patches patches/build/libs/patches-*.mpp \
  ihealth-context/iHealth+MyVitals_4.14.5.xapk
```

This produces a patched APK file (default output name is derived from the input filename). To also install immediately after patching (requires ADB connected):

```bash
java -jar /home/ben/repos/morphe-cli/build/libs/morphe-cli-1.9.1-all.jar \
  patch \
  --patches patches/build/libs/patches-*.mpp \
  -i \
  ihealth-context/iHealth+MyVitals_4.14.5.xapk
```

To enable only specific patches by name:

```bash
java -jar /home/ben/repos/morphe-cli/build/libs/morphe-cli-1.9.1-all.jar \
  patch \
  --patches patches/build/libs/patches-*.mpp \
  --exclusive -e "Disable AI broadcast modal" \
  ihealth-context/iHealth+MyVitals_4.14.5.xapk
```

### Step 6: Install the patched APK

If you didn't use `-i` above, install manually:

```bash
java -jar /home/ben/repos/morphe-cli/build/libs/morphe-cli-1.9.1-all.jar \
  utility install -a <patched-output.apk>
```

Or directly with ADB:

```bash
adb install -r <patched-output.apk>
```

### Step 7: Test on device

1. Open the iHealth app.
2. **Ad banner test**: Navigate through the app and confirm the ad banner at the bottom never appears.
3. **AI modal test**: Take a blood pressure reading and confirm the AI-generated modal with text-to-speech does not appear after the reading is complete.

If a patch doesn't work as expected, go back to JADX and look more carefully at:
- Whether the method you patched is actually the entry point, or if it's called from somewhere else
- Whether there's a visibility flag, a feature toggle, or a remote config value controlling the behavior
- Whether the same logic appears in multiple methods (might need multiple fingerprints)
